# 接口测试执行 — 前端层设计

> **读者**：Vue 3 子应用前端开发者  
> **归属**：`admin-management-station/project-sub/testgen-sub/frontend/`  
> **技术栈**：Vue 3 + Element Plus + AntV G2Plot + Pinia + Vite + Qiankun  
> **规范**：对齐 [admin-management-station](../../../admin-management-station) 子应用前端规范（`vue-web.mdc`、`subapp-layout.mdc`、`subapp-onboarding.mdc`）  
> **边界**：**不含 Agent / MCP 客户端** — 仅调用业务 BFF REST；性能分析结果来自 BFF 已落库字段  
> **参考**：[source.md](./source.md) §4.2–4.4 · [测试用例生成-前端层设计](../testgen/测试用例生成-前端层设计.md)

## 1. 定位与职责

在 **已完成的用例列表**（`TestSuitePage`）基础上，新增 **编辑**、**执行**、**监控**、**结果分析** 四类 UI 能力。

| 职责 | 说明 | 对应 source.md |
|------|------|----------------|
| 用例编辑 | 抽屉/分步表单修订 steps、http_config、断言 | §4.2 测试用例编辑组件 |
| 执行触发 | 列表单条/批量执行，选择环境与并发 | §4.2 操作栏「执行选中用例」 |
| 执行监控 | 任务状态、进度、实时指标图表、日志 | §4.3 测试执行监控页面 |
| 结果分析 | 功能/性能结果查询与可视化 | §4.4 测试结果分析页面 |
| 性能 Agent 报告展示 | 只读展示 BFF 返回的 `perf_analysis` | §6.2 性能瓶颈预测 |

**不在本次范围**：测试范围配置（`/scope`）、生成进度（`/jobs/:id`）、用例列表 CRUD 主体（已实现）。

```
用户 → Vue 页面 → testgen BFF REST (:7004/api)
                    ├─ PUT  /test-cases/:id        编辑
                    ├─ POST /test-runs             创建执行
                    ├─ GET  /test-runs/:id         状态轮询
                    └─ GET  /test-runs/:id/results 结果与 Agent 报告
监控 ← 默认 2s 轮询 GET /test-runs/:id
     ← Phase 2 可选 WS /api/ws/test-runs/:id（BFF 转发）
```

## 3. 路由增量

| 路由 | 页面 | 功能 |

## 4. 页面设计

### 4.1 TestSuitePage 增量（列表 + 入口）

在现有表格 **操作列** 扩展：

| 按钮 | 行为 |
|------|------|
| 编辑 | 打开 `TestCaseEditDrawer`，加载 `GET /test-cases/:id` |
| 执行 | 弹出 `EnvSelector` + 并发数 → `POST /test-runs` → 跳转 `/runs/:runId` |
| 删除 | **已有** |

工具栏增加：

- **执行选中**：`POST /test-runs` body `{ case_ids: [], env_id, concurrency }`
- 批量执行时 `runId` 为 batch run，监控页展示子任务列表

### 4.2 TestCaseEditDrawer — 用例编辑

参考 source.md §4.2 四步抽屉，字段对齐 **testgen 已有模型** + **HTTP 执行扩展**：

| 步骤 | 字段 | 组件 | 映射 DB |
|------|------|------|---------|
| 基础信息 | title, module, type, priority, tags | el-form | `test_cases.*` |
| 请求参数 | method, url, headers, body | HttpConfigForm | `http_config` JSONB |
| 断言规则 | status_code, body_path, latency_ms | AssertRulesEditor | `http_config.assertions` |
| 步骤与预期 | steps[], expected, preconditions | 已有 ExpandableTextBlock 风格 | `steps`, `expected` |

**保存**：`PUT /api/test-cases/:id`，成功后关闭抽屉并 `suiteStore.refresh()`。

**校验**：

- `title` 必填；`http_config.method` ∈ GET/POST/PUT/DELETE/PATCH
- 断言至少一条或保留空（功能测试可仅验 status）

### 4.3 TestRunMonitorPage — 执行监控

布局参考 source.md §4.3，简化为 **2×2 卡片 + 任务表**：

| 区域 | 内容 | 数据源 |
|------|------|--------|
| 状态概览 | 运行中 / 成功 / 失败计数 | `GET /test-runs/:id` |
| 性能指标 | 响应时间折线（G2Plot Line） | 轮询 `metrics` 或 WS |
| 实时日志 | el-scrollbar 文本 | `run.log_tail` |
| 任务列表 | 子 run 或单 run 详情 | `run.items[]` |

**composables/useTestRunProgress.js**：

```javascript
// 默认 2s 轮询；status ∈ done|failed|cancelled 时停止
export function useTestRunProgress(runId) {
  const store = useTestRunStore();
  onMounted(() => store.startPolling(runId));
  onUnmounted(() => store.stopPolling());
  watch(() => store.status, (s) => {
    if (['success', 'failed', 'cancelled'].includes(s)) {
      router.push({ name: 'test-run-results', params: { runId } });
    }
  });
  return { store };
}
```

**Phase 2 WebSocket**（`VITE_ENABLE_RUN_WS=true`）：连接 BFF `/api/ws/test-runs/:id`，消息 `{ type: 'STATUS_UPDATE', progress, metrics }` — **禁止**直连 Agent 或 `:4001`。

### 4.4 TestResultAnalysisPage — 结果分析

| 区块 | 说明 |
|------|------|
| 查询条件 | runId（路由参数）、环境、测试类型 filter |
| 功能结果表 | request_id、status、response_time、http_status、error |
| 性能图表 | TPS、P95、错误率（G2Plot） |
| PerfAnalysisPanel | 展示 `run.perf_analysis`：瓶颈列表、优化建议、风险等级 |

性能测试 **且** `perf_analysis.status === 'done'` 时展示 Agent 报告；否则显示「分析中」或「未触发（非性能 run）」。

## 5. API 对接（仅 BFF）

### 5.1 testRunService.js

| 方法 | BFF 路径 | 说明 |
|------|----------|------|
| `createRun(payload)` | `POST /test-runs` | `{ case_ids, env_id, mode, concurrency }` |
| `getRun(id)` | `GET /test-runs/:id` | 状态、progress、metrics 摘要 |
| `cancelRun(id)` | `POST /test-runs/:id/cancel` | 取消 RUNNING |
| `getRunResults(id)` | `GET /test-runs/:id/results` | func + perf + perf_analysis |
| `listRuns(params)` | `GET /test-runs` | Phase 2 执行历史 |

### 5.2 envConfigService.js

| 方法 | BFF 路径 | 说明 |
|------|----------|------|
| `listEnvs()` | `GET /env-configs` | 下拉选项 |
| `createEnv(body)` | `POST /env-configs` | Phase 2 环境管理页 |

### 5.3 testCaseService.js 扩展

`updateTestCase` 请求体增加：

```javascript
{
  title, module, type, priority, preconditions, steps, expected, tags,
  http_config: {
    method: 'POST',
    url: '/api/booking',
    headers: {},
    body: {},
    assertions: [
      { type: 'status', expect: 200 },
      { type: 'json_path', path: '$.code', expect: 0 },
      { type: 'latency_ms', max: 500 },
    ],
  },
}
```

  state: () => ({
    runId: null,

  getters: {
    isTerminal: (s) => ['success', 'failed', 'cancelled'].includes(s.status),

## 9. 与 source.md / testgen 的差异

| source.md 原方案 | 本设计（优化） |
|------------------|----------------|
| 独立「测试用例管理」整页 | 复用 TestSuitePage，仅增编辑/执行 |
| 前端 WS 直连 `:7001/websocket/test-run` | 轮询 BFF；可选 BFF 转发 WS |
| Element Plus 全局 G2Plot 注册 | 按需 import 图表组件 |
| 新建用例流程 | **剔除**（生成链路在 testgen） |
| callAI 前端调用 | **禁止**；perf 报告只读 BFF 字段 |
